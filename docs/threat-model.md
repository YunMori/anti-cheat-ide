# 초기 위협 모델

## 자산(Assets)

- 시험 콘텐츠 및 숨은 테스트 케이스
- 응시자 소스 코드 및 제출물
- 세션 행동 이벤트
- 채점(Judge) 인프라
- 검토자 결정 및 감사 로그

## 신뢰 경계(Trust boundaries)

1. 응시자 브라우저는 신뢰할 수 없다.
2. Platform API는 모든 브라우저 입력을 검증한다.
3. Judge Service는 모든 소스 제출물을 적대적인 것으로 취급한다.
4. Detection 근거는 불완전하거나 조작되었을 수 있으며 참고용(advisory)이다.
5. 관리자 접근에는 더 강한 인가와 감사 로깅이 필요하다.

## 초기 위협과 대응책

| 위협 | 초기 대응책 |
| --- | --- |
| 대량 붙여넣기 또는 생성된 답안 | 붙여넣기 및 code-burst 신호와 검토자용 근거 |
| 자동화된 인간형 타이핑 | 타이핑 규칙성, 코드 변경, 포커스 신호 결합 |
| 위조된 브라우저 이벤트 | 순서 보장 배치, 중복 제거, 서버 수신 타임스탬프 |
| 무한 루프 및 자원 고갈 | Judge의 벽시계(wall-clock) 및 자원 제한 |
| 샌드박스 탈출 | 컨테이너 격리, 네트워크 차단, 최소 런타임 이미지 |
| 숨은 테스트 케이스 유출 | Judge 전용 접근 및 클라이언트 미전달 |
| 잘못된 부정행위 누명 | 다중 신호 검토 및 자동 탈락 금지 |
| 이벤트 데이터 프라이버시 침해 | 최소 수집 및 명시적 보존 정책 |

## 보류된 대응책(Deferred controls)

- 하드웨어 기반 브라우저 증명(attestation)
- 조직 SSO 및 고급 RBAC
- 제출물 간 구조적 유사도 분석
- Firecracker 또는 gVisor 샌드박싱
- 동의 기반 운영 데이터로 보정된(calibrated) ML 위험 점수
